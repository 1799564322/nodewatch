这份文档用于帮助项目作者准确表达真实工作，不应直接背诵，也不要写入没有验证过的用户量或性能数据。

## 简历项目描述（精简版）

**NodeWatch｜轻量级跨平台主机监控平台｜个人全栈项目**

- 使用 Python Agent 采集 Windows/Linux 主机的 CPU、内存、磁盘、网络和系统信息，通过独立设备 Token 与 HTTPS 上报至 FastAPI 服务端。
- 使用 Vue 3、TypeScript、Element Plus 和 ECharts 实现设备总览、历史指标、多磁盘及告警面板；PostgreSQL 同时保存历史样本与 latest 快照，并在数据库端完成时间桶聚合。
- 为 Agent 实现 SQLite WAL 离线缓存、指数退避、幂等批量补传和 Windows/Linux 独立打包；使用 GitHub Actions、GHCR、Docker Compose、Nginx 完成单机部署、HTTPS、备份恢复和版本回滚。

可按版面删减，但只保留自己能够从代码、测试和部署记录中解释的内容。

## 30 秒介绍

NodeWatch 是我从需求、开发、测试到部署完成的个人全栈项目。被监控电脑运行一个跨平台 Python Agent，指标先写入本地 SQLite，再通过 HTTPS 批量发送到 FastAPI；服务端用 PostgreSQL 保存历史和最新状态，Vue 页面展示曲线与告警。项目部署在 2 核 2 GiB 云服务器上，并完成了自动构建、HTTPS、备份恢复、回滚以及 Windows/Linux 安装验证。

## 2 分钟讲解顺序

1. **问题**：需要低成本查看多台个人设备的状态，且网络中断时不能静默丢样本。
2. **数据流**：Agent 采集 → SQLite 持久队列 → HTTPS 幂等批量 API → PostgreSQL 历史/latest → Vue 页面。
3. **可靠性**：采集先落盘，服务端确认后才删队列；网络错误指数退避；旧补传样本不覆盖最新值。
4. **告警**：实时阈值与定时离线检查组成状态机，支持持续时间、去重、确认、恢复和维护模式。
5. **交付**：CI 测试、固定 tag 镜像、单机 Compose、Nginx TLS、数据库备份恢复和 App 独立回滚。
6. **边界**：目标规模是 10～50 台，并非高可用；当前告警只在网页展示。

## 高频面试问题

### 为什么要有 latest 表？

设备列表每次只需要最新状态。若从持续增长的历史表为每台设备找最后一条，查询会越来越重。写入时维护一行 latest，换取稳定简单的读取；同时用采集时间判断，避免离线补传的旧数据覆盖新状态。

### 为什么离线由服务端判断？

离线设备本身无法发送“我离线了”。服务端根据最后上报时间统一判断，才能发现断电、断网和 Agent 退出，并能结合维护模式排除预期离线。

### 为什么 Agent 先写 SQLite？

内存队列会在进程退出或机器重启时丢失。SQLite 是跨平台单文件数据库，事务和 WAL 能以较低复杂度持久保存待传样本。只有服务端确认 accepted 或 duplicate 后才删除，形成 at-least-once 传输配合服务端幂等。

### 如何保证重复补传不产生重复数据？

服务端以设备 ID 和采集时间作为业务唯一性约束。重复样本返回 duplicate，Agent 收到确认后可安全删除本地记录。

### 为什么旧样本不触发实时告警？

断网恢复时可能补传数小时前的数据。如果用它触发实时告警，用户会收到已经过期的事件；因此历史仍保存，但 latest 更新和实时告警只接受比现有 latest 更新的样本。

### 为什么生产只能一个 Worker？

APScheduler 目前运行在应用进程内。多 Worker 会各自执行离线检查和清理任务。当前目标规模下固定单 Worker 更简单；横向扩容前应拆出独立调度服务或增加分布式锁。

### 2 核 2 GiB 为什么够用？

项目明确限定 10～50 台、60 秒采样，前后端合并为一个 App 容器，数据库不额外公开，且历史查询会聚合并限制点数。这是针对目标规模的工程选择，不代表未经压测的更高容量承诺。

### 生产中遇到过什么问题？

- 阿里云访问 Docker Hub 超时：由 CI 把固定版本官方 PostgreSQL 镜像同步到 GHCR，避免使用未知第三方镜像站。
- Windows 覆盖安装时旧进程占用 exe：安装脚本先注销任务、结束安装目录中的旧进程并等待解锁，再替换程序。
- 无域名但需要受信任 HTTPS：使用支持公网 IP 的短期证书，保留 HTTP-01 路径并验证 Certbot 自动续期钩子。
- 公网恢复连接后有大量缓存：服务端限制批次，Agent 分批上传，最终 `remaining=0`，历史按原采集时间写入。

## 可以主动说明的改进方向

- 将调度器拆成独立进程，支持 App 横向扩容。
- 增加登录限速、MFA、审计日志查询和通知渠道。
- 为历史表增加分区或归档策略，并用真实压力测试确定容量边界。
- 增加 Prometheus/OpenTelemetry 自监控和集中日志。
- 将备份加密后复制到独立对象存储并配置失败告警。

## AI 辅助开发如何诚实表达

可以说使用 AI 辅助生成样板、排查日志和完善测试，但需求取舍、运行验证、服务器操作、安全检查和最终验收由自己完成。面试时应能定位关键代码、解释数据库约束、复述故障原因，并现场说明为什么采用当前方案。
