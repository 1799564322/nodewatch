# NodeWatch 技术决策

## D-001：P0 保持最小可运行骨架

- 状态：已采纳
- 决策：P0 后端只提供不访问数据库的 `/api/v1/health/live`，前端只提供占位页。
- 原因：遵守阶段边界，避免提前实现 P1 的数据库、就绪检查和登录。

## D-002：开发环境数据库使用独立 Compose 文件

- 状态：已采纳
- 决策：`deploy/docker-compose.local.yml` 只运行 PostgreSQL 17，并仅供本地开发。
- 原因：让初学者能单独理解和控制数据库生命周期，不混入尚未实现的应用镜像。

## D-003：统一使用 Python 3.12

- 状态：已采纳
- 决策：Agent、后端和 CI 均以 Python 3.12 为基线。
- 原因：与交付文档及最终部署环境保持一致。

## D-004：前端类型检查不写增量缓存

- 状态：已采纳
- 决策：使用 `vue-tsc --noEmit` 和 `tsc --noEmit` 分别检查应用与工具配置。
- 原因：P0 项目规模很小，不需要增量编译缓存；无输出检查也更适合 CI 和受限开发环境。

## D-005：P1 使用数据库 Session，不引入 Redis

- 状态：已采纳
- 决策：随机 Session Token 通过 HttpOnly Cookie 交给浏览器，数据库只保存 Token 的 SHA-256 哈希。
- 原因：满足服务端撤销登录的需求，同时减少 P1 的基础设施数量，更适合 2 核 2G 服务器和初学者理解。

## D-006：数据库结构必须通过 Alembic 迁移

- 状态：已采纳
- 决策：应用启动时不调用 `create_all`，数据库表通过 `alembic upgrade head` 创建和升级。
- 原因：迁移文件可审查、可重复执行，并能明确记录数据库结构的演进过程。

## D-007：首个管理员只在空用户表中初始化

- 状态：已采纳
- 决策：启动时仅当 `users` 表为空才读取 Bootstrap 配置创建管理员。
- 原因：避免每次启动覆盖既有账号或密码，同时保留首次部署的简单流程。

## D-008：Agent Token 与本地随机实例 UUID 绑定

- 状态：已采纳
- 决策：Agent 首次运行持久化随机 UUID，Token 首次 bootstrap 后绑定该 UUID；不使用 MAC 地址或硬盘序列号。
- 原因：避免硬件变化和隐私问题，并防止复制出的 Token 被另一 Agent 实例抢占。

## D-009：P2 不保存 CPU/内存历史

- 状态：已采纳
- 决策：Agent 已能采集并在本地日志展示 CPU/内存，但服务端 P2 只接收系统信息。
- 原因：严格遵守阶段边界；历史样本、latest 表和指标 API 属于 P3。

## D-010：历史指标与最新指标分表保存

- 状态：已采纳
- 决策：每个样本写入 `metric_samples`，同时按采集时间 upsert `device_latest_metrics`。
- 原因：历史数据适合时间范围查询，latest 表适合总览和设备列表；旧补传样本不能覆盖较新的当前状态。

## D-011：历史降采样在 PostgreSQL 中完成

- 状态：已采纳
- 决策：使用 `date_bin` 对 5 分钟和 1 小时桶求平均，API 最多返回 1500 点。
- 原因：减少数据库到后端及浏览器的数据传输，并让前端只负责展示，不承担统计逻辑。

## D-012：告警调度固定单进程

- 状态：已采纳
- 决策：MVP 使用进程内 APScheduler，部署时保持单副本和单 Worker。
- 原因：适合 2 核 2G 单机规模；在引入独立调度器或分布式锁前，多 Worker 会重复执行离线检查和清理任务。

## D-013：Agent 使用 SQLite WAL 持久化待上报指标

- 状态：已采纳
- 决策：Agent 采集后先写本地 SQLite，按采集时间批量上传，服务端确认接受或重复后才删除；默认最多 10000 条并保留 7 天。
- 原因：单文件数据库适合 Windows/Linux 独立 Agent，不需要额外服务；WAL、事务和容量上限兼顾异常退出恢复与磁盘可控。

## D-014：Agent 重试按错误类型分级

- 状态：已采纳
- 决策：网络、429 和 5xx 使用带抖动的指数退避；401、403、409 及其他不可重试错误等待 300 秒后再检查。
- 原因：临时故障恢复后可自动补传，同时避免错误 Token 或绑定冲突每秒请求服务端。

## D-015：生产环境由单镜像同源提供 Vue 与 FastAPI

- 状态：已采纳
- 决策：多阶段 Dockerfile 构建前端并复制到后端静态目录，Nginx 只反向代理一个 App 服务。
- 原因：避免额外前端容器和跨域配置，降低 2 核 2 GiB 单机部署的内存与运维复杂度。

## D-016：生产数据库仅使用 Compose 内部网络

- 状态：已采纳
- 决策：PostgreSQL 不配置 `ports`，App 只映射到宿主机 `127.0.0.1:8000`，公网流量必须经过 Nginx。
- 原因：数据库和 Uvicorn 无需直接暴露公网，可显著缩小攻击面，同时仍保留宿主机本地诊断能力。

## D-017：无域名部署使用受信任的短期公网 IP 证书

- 状态：已采纳
- 决策：项目不配置域名，由 Nginx 使用支持公网 IP 的 Let’s Encrypt 短期证书提供 HTTPS，保留 HTTP-01 验证路径并启用 Certbot 自动续期。
- 原因：满足项目所有者“不配置域名”与生产 Cookie 必须走可信 HTTPS 的共同约束；短有效期证书要求持续验证续期定时器和 reload 钩子。

## D-018：演示数据必须与生产和真实设备隔离

- 状态：已采纳
- 决策：seed 只操作固定 UUID、带 `[演示]` 前缀的数据，UUID 名称冲突时中止，并在 `APP_ENV=production` 时拒绝执行。
- 原因：简历截图需要稳定可重复的数据，但不能为了演示污染生产指标或误删真实设备。
